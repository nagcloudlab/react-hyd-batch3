Component Pattern

Higher-Order Components (HOC)

A function that takes a component and returns a new enhanced component

The Problem

Cross-Cutting Concerns Lead to Duplicated Logic

Multiple components need the same behavior — authentication checks, logging, theming, data fetching. Without a reuse strategy, you end up copy-pasting the same logic into every component. When that logic changes, you have to update it everywhere.

Before: Duplicated Auth Checks Everywhere

import { useState, useEffect } from 'react';
import { getCurrentUser } from './auth';

// --- Dashboard.jsx ---
function Dashboard() {
  const [user, setUser] = useState(null);
  const [loading, setLoading] = useState(true);

  useEffect(() => {
    getCurrentUser()
      .then(u => setUser(u))
      .finally(() => setLoading(false));
  }, []);

  if (loading) return <p>Loading...</p>;
  if (!user) return <Redirect to="/login" />;

  return <h1>Welcome to Dashboard, {user.name}</h1>;
}

// --- Settings.jsx --- (same auth logic, again)
function Settings() {
  const [user, setUser] = useState(null);
  const [loading, setLoading] = useState(true);

  useEffect(() => {
    getCurrentUser()
      .then(u => setUser(u))
      .finally(() => setLoading(false));
  }, []);

  if (loading) return <p>Loading...</p>;
  if (!user) return <Redirect to="/login" />;

  return <h1>Settings for {user.name}</h1>;
}

// --- Profile.jsx --- (and again!)
function Profile() {
  const [user, setUser] = useState(null);
  const [loading, setLoading] = useState(true);

  useEffect(() => {
    getCurrentUser()
      .then(u => setUser(u))
      .finally(() => setLoading(false));
  }, []);

  if (loading) return <p>Loading...</p>;
  if (!user) return <Redirect to="/login" />;

  return <h1>Profile: {user.name}</h1>;
}

Three components. Identical auth logic in each. If you need to change the redirect path or add role-based checks, you have to touch every file. This does not scale.

How a Higher-Order Component Works

A Higher-Order Component is a function that accepts a component and returns a new component with additional behavior. The original component is never modified — it is wrapped and composed.

The Solution

Create a Reusable withAuth HOC

Write the auth logic once inside a wrapper function. Any component you pass in automatically gains authentication behavior, loading states, and redirect handling — without changing a single line of its own code.

The HOC Implementation

import { useState, useEffect } from 'react';
import { Navigate } from 'react-router-dom';
import { getCurrentUser } from './auth';

// The Higher-Order Component
function withAuth(WrappedComponent) {
  // Return a new component
  return function WithAuthComponent(props) {
    const [user, setUser] = useState(null);
    const [loading, setLoading] = useState(true);

    useEffect(() => {
      getCurrentUser()
        .then(u => setUser(u))
        .finally(() => setLoading(false));
    }, []);

    if (loading) return <p>Authenticating...</p>;
    if (!user) return <Navigate to="/login" />;

    // Pass the user + original props to the wrapped component
    return <WrappedComponent user={user} {...props} />;
  };
}

// Give it a useful display name for React DevTools
// (see "Best Practices" below)

After: Clean, DRY Components

// --- Dashboard.jsx ---
function Dashboard({ user }) {
  return <h1>Welcome to Dashboard, {user.name}</h1>;
}
export default withAuth(Dashboard);

// --- Settings.jsx ---
function Settings({ user }) {
  return <h1>Settings for {user.name}</h1>;
}
export default withAuth(Settings);

// --- Profile.jsx ---
function Profile({ user }) {
  return <h1>Profile: {user.name}</h1>;
}
export default withAuth(Profile);

// Auth logic lives in ONE place. Components stay focused on their own UI.

When to Use

Use HOCs When

  • You have cross-cutting concerns shared by many components (auth, logging, analytics, theming)
  • You're working with class components or a legacy codebase where hooks aren't an option
  • You need to conditionally render a component (e.g., show a login page instead)
  • You want to inject props transparently without the wrapped component knowing

Avoid HOCs When

  • You're using function components and a custom hook would be simpler
  • Stacking multiple HOCs creates "wrapper hell" in the component tree
  • The HOC needs to pass many props, causing prop name collisions
  • The abstraction obscures what props a component receives, hurting readability

HOCs in the Wild

Higher-Order Components were the dominant pattern for logic reuse before React Hooks arrived in version 16.8. You will still encounter them frequently in production codebases:

  • connect(mapState, mapDispatch)(Component) — Redux's classic API for connecting components to the store
  • withRouter(Component) — React Router v5 for injecting history, location, and match props
  • withStyles(styles)(Component) — Material UI's styling injection (JSS-based)
  • memo(Component) — React's own React.memo is technically a HOC that adds memoization

Best Practice: Set a Display Name

function withAuth(WrappedComponent) {
  function WithAuthComponent(props) {
    // ... auth logic
  }

  // Helps identify components in React DevTools
  WithAuthComponent.displayName =
    `WithAuth(${WrappedComponent.displayName || WrappedComponent.name || 'Component'})`;

  return WithAuthComponent;
}

Key Takeaway

HOCs follow the principle: don't modify the original component, compose it. Think of them as decorators for React components. They wrap behavior around a component without touching its internals — a clean application of the open/closed principle.
