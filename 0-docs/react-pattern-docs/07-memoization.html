Performance Pattern

Memoization

Prevent unnecessary re-renders and expensive recalculations with React.memo, useMemo, and useCallback

The Problem

Unnecessary Work on Every Render

In React, when a parent re-renders, all its children re-render too — even if their props haven't changed. Expensive computations run again on every render even when their inputs are identical. Functions are recreated on every render, breaking shallow equality checks for child components.

Consider this common scenario — clicking the counter re-renders the expensive list even though it has nothing to do with the count:

function App() {
  const [count, setCount] = useState(0);
  const [items] = useState(generateLargeList()); // 10,000 items

  // This runs on EVERY render, even when items haven't changed
  const sorted = items
    .filter(item => item.active)
    .sort((a, b) => a.name.localeCompare(b.name));

  // New function reference every render
  const handleSelect = (id) => console.log('Selected:', id);

  return (
    <div>
      <button onClick={() => setCount(c => c + 1)}>
        Count: {count}   {/* Clicking this re-renders EVERYTHING below */}
      </button>

      {/* This re-renders on every count change, for no reason */}
      <ExpensiveList items={sorted} onSelect={handleSelect} />
    </div>
  );
}

Re-render Decision Tree

The Solution: Three Memoization Tools

React.memo — Memoize Components

Wrap a component with React.memo() to skip re-rendering when its props haven't changed (shallow comparison).

// Before: re-renders every time parent renders, even with same props
function ExpensiveList({ items, onSelect }) {
  console.log('ExpensiveList rendered!'); // Fires on every parent render
  return (
    <ul>
      {items.map(item => (
        <li key={item.id} onClick={() => onSelect(item.id)}>
          {item.name}
        </li>
      ))}
    </ul>
  );
}

// After: only re-renders when items or onSelect actually change
const ExpensiveList = React.memo(function ExpensiveList({ items, onSelect }) {
  console.log('ExpensiveList rendered!'); // Only fires when props change
  return (
    <ul>
      {items.map(item => (
        <li key={item.id} onClick={() => onSelect(item.id)}>
          {item.name}
        </li>
      ))}
    </ul>
  );
});

useMemo — Memoize Expensive Values

Cache the result of an expensive computation so it only recalculates when its dependencies change.

function App({ items, filter }) {
  // Before: sorts and filters on EVERY render
  const sorted = items
    .filter(item => item.category === filter)
    .sort((a, b) => a.name.localeCompare(b.name));

  // After: only recalculates when items or filter change
  const sorted = useMemo(() => {
    return items
      .filter(item => item.category === filter)
      .sort((a, b) => a.name.localeCompare(b.name));
  }, [items, filter]); // Dependency array: recalculate only when these change

  return <ItemList items={sorted} />;
}

useCallback — Memoize Functions

Stabilize a function reference so memoized children don't re-render just because the function was recreated.

function App() {
  const [count, setCount] = useState(0);
  const [items] = useState(generateLargeList());

  // Before: new function on every render => breaks React.memo on child
  const handleSelect = (id) => {
    console.log('Selected:', id);
  };

  // After: stable function reference, only recreated when dependencies change
  const handleSelect = useCallback((id) => {
    console.log('Selected:', id);
  }, []); // Empty deps = never recreated

  return (
    <div>
      <button onClick={() => setCount(c => c + 1)}>
        Count: {count}
      </button>
      {/* Now this truly skips re-rendering when count changes */}
      <ExpensiveList items={items} onSelect={handleSelect} />
    </div>
  );
}

Putting It All Together

const ExpensiveList = React.memo(({ items, onSelect }) => {
  return (
    <ul>
      {items.map(item => (
        <li key={item.id} onClick={() => onSelect(item.id)}>{item.name}</li>
      ))}
    </ul>
  );
});

function App() {
  const [count, setCount] = useState(0);
  const [filter, setFilter] = useState('all');
  const [items] = useState(generateLargeList());

  // useMemo: only recalculate when items or filter change
  const filteredItems = useMemo(() => {
    return filter === 'all'
      ? items
      : items.filter(i => i.category === filter);
  }, [items, filter]);

  // useCallback: stable function reference
  const handleSelect = useCallback((id) => {
    console.log('Selected:', id);
  }, []);

  return (
    <div>
      {/* Changing count does NOT re-render ExpensiveList */}
      <button onClick={() => setCount(c => c + 1)}>Count: {count}</button>
      <select onChange={e => setFilter(e.target.value)}>...</select>
      {/* React.memo: skip re-render if filteredItems and handleSelect are same */}
      <ExpensiveList items={filteredItems} onSelect={handleSelect} />
    </div>
  );
}

Don't Memoize Everything!

Premature Optimization is the Root of All Evil

Memoization is not free. Every useMemo and useCallback has a cost: memory to store the cached value, and a comparison check on every render. If the computation is trivial or the component always receives new props, memoization adds overhead with no benefit.

  • Profile first. Use React DevTools Profiler to identify actual bottlenecks.
  • React's re-renders are fast by default. Most components don't need memoization.
  • Memoization has a cost: memory for cached values + shallow comparison on every render.
  • If props always change, React.memo makes things slower (comparison + re-render).

When to Use

Use When

  • Expensive computations (sorting, filtering large arrays)
  • Large lists or complex component trees
  • Components that render frequently with the same props
  • Passing callbacks to memoized children
  • You've profiled and confirmed a bottleneck

Avoid When

  • Simple, cheap-to-render components
  • Premature optimization without profiling
  • Components that always receive new props
  • Primitive value computations (a + b)
  • You haven't measured a performance problem

Key Takeaway

Memoization is a tool, not a default. Profile first, identify the bottleneck, then apply the right memoization tool. React.memo for components, useMemo for values, useCallback for functions. Remember: React is fast by default — only optimize when you have evidence of a problem.
