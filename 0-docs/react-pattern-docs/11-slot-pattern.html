Component Pattern

Slot Pattern

Create flexible, composable layouts by letting consumers inject content into predefined slots

Inspired by Web Component <slot> elements and Vue.js named slots, the Slot Pattern adapts the same idea for React using props and children. It solves one of the most common design system challenges: building reusable layout components that are structured yet flexible.

The Problem

Prop Explosion in Layout Components

You're building a reusable Card, Dialog, or AppShell. It has a fixed structure — header, body, footer — but consumers need to customize what goes in each area. Using a single children prop doesn't give enough control. You end up with either:

  • Too many string/element props (title, subtitle, headerIcon, headerAction, footerLeft, footerRight…) — the dreaded "prop explosion" problem.
  • Or a rigid component that can't be customized at all, forcing consumers to build their own from scratch.

Before: A Card with Prop Explosion

// Card with 8+ individual props for every customizable area
function Card({
  title,
  subtitle,
  headerIcon,
  headerAction,
  badge,
  badgeColor,
  children,
  footerPrimaryLabel,
  footerPrimaryOnClick,
  footerSecondaryLabel,
  footerSecondaryOnClick,
  footerAlign,
  showDivider,
}) {
  return (
    <div className="card">
      <div className="card-header">
        {headerIcon && <span className="icon">{headerIcon}</span>}
        <div>
          {badge && <span style={{ color: badgeColor }}>{badge}</span>}
          {title && <h3>{title}</h3>}
          {subtitle && <p>{subtitle}</p>}
        </div>
        {headerAction && <button>{headerAction}</button>}
      </div>
      {showDivider && <hr />}
      <div className="card-body">{children}</div>
      <div className="card-footer" style={{ justifyContent: footerAlign }}>
        {footerSecondaryLabel && (
          <button onClick={footerSecondaryOnClick}>{footerSecondaryLabel}</button>
        )}
        {footerPrimaryLabel && (
          <button onClick={footerPrimaryOnClick}>{footerPrimaryLabel}</button>
        )}
      </div>
    </div>
  );
}

// Every new customization need = another prop = another update to Card
// What if someone wants TWO badges? A dropdown in the header? A link in the footer?

How Slots Work

The Solution

Named Props as Slots

Instead of many granular props, accept a few slot props that take JSX elements. This gives consumers full control over what goes in each area while your component retains control over structure and layout.

Approach 1: Props-Based Slots (Most Common)

The simplest and most widely used approach. Each slot is a prop that accepts a ReactNode. 

// A clean Card with slot props instead of prop explosion
function Card({ header, children, footer }) {
  return (
    <div className="card">
      {header && <div className="card-header">{header}</div>}
      <div className="card-body">{children}</div>
      {footer && <div className="card-footer">{footer}</div>}
    </div>
  );
}

// Usage — consumers have FULL control over each slot's content
<Card
  header={
    <>
      <Avatar src={user.pic} />
      <h3>{user.name}</h3>
    </>
  }
  footer={<Button>Save</Button>}
>
  <p>Card body content here</p>
</Card>

Approach 2: Compound Slots (Using Sub-Components)

For richer APIs, attach slot components directly to the parent. This reads like declarative HTML and scales well for complex layouts. 

import React, { Children, isValidElement } from 'react';

// Helper: find a specific slot child by type
function findSlot(children, slotType) {
  return Children.toArray(children).find(
    (child) => isValidElement(child) && child.type === slotType
  );
}

// Helper: get everything that ISN'T a named slot
function filterSlots(children, slotTypes) {
  return Children.toArray(children).filter(
    (child) => !isValidElement(child) || !slotTypes.includes(child.type)
  );
}

function Page({ children }) {
  const header  = findSlot(children, Page.Header);
  const sidebar = findSlot(children, Page.Sidebar);
  const footer  = findSlot(children, Page.Footer);
  const content = filterSlots(children, [
    Page.Header, Page.Sidebar, Page.Footer
  ]);

  return (
    <div className="page-layout">
      <header>{header}</header>
      <aside>{sidebar}</aside>
      <main>{content}</main>
      <footer>{footer}</footer>
    </div>
  );
}

// Define slot sub-components
Page.Header  = ({ children }) => children;
Page.Sidebar = ({ children }) => children;
Page.Footer  = ({ children }) => children;

// Usage — reads like declarative HTML
<Page>
  <Page.Header>
    <Logo /> <Nav />
  </Page.Header>
  <Page.Sidebar>
    <MenuItems />
  </Page.Sidebar>
  Main content here
  <Page.Footer>
    <Copyright />
  </Page.Footer>
</Page>

Approach 3: Function Slots (When Slots Need Parent Data)

When slots need to receive data from the parent component, use function slots — a fusion of the Slot and Render Props patterns. 

function DataTable({ data, columns, headerSlot, rowSlot, emptySlot }) {
  if (data.length === 0) {
    return emptySlot?.() || <p>No data</p>;
  }

  return (
    <table>
      <thead>
        {headerSlot?.(columns) || (
          <tr>
            {columns.map(col => <th key={col.key}>{col.label}</th>)}
          </tr>
        )}
      </thead>
      <tbody>
        {data.map((row, i) =>
          rowSlot?.(row, i) || <DefaultRow key={i} row={row} />
        )}
      </tbody>
    </table>
  );
}

// Usage — slots receive data from parent
<DataTable
  data={users}
  columns={[
    { key: 'name', label: 'Name' },
    { key: 'role', label: 'Role' },
  ]}
  headerSlot={(cols) => (
    <tr>
      {cols.map(c => <th key={c.key} style={{ color: 'blue' }}>{c.label}</th>)}
    </tr>
  )}
  rowSlot={(user, index) => (
    <tr key={user.id} className={index % 2 ? 'striped' : ''}>
      <td><UserAvatar user={user} /> {user.name}</td>
      <td><RoleBadge role={user.role} /></td>
    </tr>
  )}
  emptySlot={() => (
    <EmptyState icon="users" message="No users found" />
  )}
/>

Full Example: AppShell with Slots

A realistic layout component for an entire application page, using props-based slots. 

function AppShell({ navbar, sidebar, children, footer }) {
  return (
    <div className="app-shell">
      {navbar && (
        <nav className="app-navbar">{navbar}</nav>
      )}
      <div className="app-body">
        {sidebar && (
          <aside className="app-sidebar">{sidebar}</aside>
        )}
        <main className="app-content">{children}</main>
      </div>
      {footer && (
        <footer className="app-footer">{footer}</footer>
      )}
    </div>
  );
}

// Usage in your app
<AppShell
  navbar={
    <div className="flex items-center gap-4">
      <Logo />
      <SearchBar />
      <UserMenu user={currentUser} />
    </div>
  }
  sidebar={
    <Navigation items={menuItems} active={currentPath} />
  }
  footer={
    <p>&copy; 2026 My App. All rights reserved.</p>
  }
>
  {/* This is the main content area */}
  <PageHeader title="Dashboard" />
  <StatsGrid data={stats} />
  <RecentActivity items={activities} />
</AppShell>

TypeScript Typing for Slots

import { ReactNode } from 'react';

interface CardProps {
  header?: ReactNode;      // Optional slot
  children: ReactNode;     // Required slot
  footer?: ReactNode;      // Optional slot
}

interface DataTableProps<T> {
  data: T[];
  columns: Column[];
  headerSlot?: (columns: Column[]) => ReactNode;  // Function slot
  rowSlot?: (item: T, index: number) => ReactNode; // Function slot
  emptySlot?: () => ReactNode;                     // Function slot
}

Comparing Slot Approaches

Approach Pros Cons
Props-based slots Simple, explicit, easy to type with TypeScript Slots can't communicate with each other easily
Compound slots Clean JSX, feels like HTML, scales well More complex implementation, requires helper functions
Function slots Slots receive data from the parent component Less readable for simple cases, blurs into render props

Real-World Examples

The Slot Pattern is everywhere in production React libraries:

  • Next.js Layout pattern — layout.tsx receives children as a slot for nested routes. Parallel routes like @modal are additional named slots.
  • Ant Design's Card — accepts title, extra, cover, and actions as slot props alongside children.
  • Material UI — AppBar, Card, and Dialog all use sub-components as compound slots (e.g., CardHeader, CardContent, CardActions).
  • Radix UI's Dialog — uses compound slots: Dialog.Trigger, Dialog.Content, Dialog.Title, Dialog.Description, Dialog.Close.

When to Use

Use When

  • Building reusable layout components (Card, Dialog, Page, AppShell, Modal)
  • Creating design system components where structure is fixed but content varies
  • A component has 3+ customizable areas and string props aren't expressive enough
  • You want consumers to inject arbitrary JSX — not just strings or simple values

Don't Use When

  • A component has only 1–2 customization points — just use children and maybe one prop
  • The structure itself needs to change — slots assume a fixed layout
  • Simple text customization is enough — a title: string prop is simpler than a slot for a heading
  • You need bidirectional communication between slots — consider Context or State Reducer instead

Key Takeaway

The Slot Pattern turns your components into flexible templates. Instead of predicting every customization need with individual props, provide named slots and let consumers fill them with whatever they need. Think of it as creating a picture frame — you provide the frame, they provide the picture.
