Logic Pattern

State Reducer Pattern

Give component consumers full control over state changes

The Problem

The Endless Props Trap

You build a reusable Toggle component. Then one consumer needs a toggle that can only be toggled 4 times. Another needs a toggle that can't be turned off. A third needs one that resets after 10 seconds. You keep adding one-off props until your component is drowning in special cases.

Here's what this looks like in practice — a Toggle component that has grown out of control:

// The Toggle component that tries to handle every use case via props
function Toggle({
  onToggle,
  maxClicks,          // Consumer A needs this
  preventOff,         // Consumer B needs this
  preventOn,          // Consumer C needs this
  resetAfter,         // Consumer D needs this
  confirmBeforeOff,   // Consumer E needs this
  disableAfterFirst,  // Consumer F needs this
  cooldownMs,         // Consumer G needs this
  // ...and the list keeps growing
}) {
  const [on, setOn] = useState(false);
  const [clickCount, setClickCount] = useState(0);

  function handleToggle() {
    if (maxClicks && clickCount >= maxClicks) return;
    if (preventOff && on) return;
    if (preventOn && !on) return;
    if (confirmBeforeOff && on && !window.confirm('Turn off?')) return;
    // ...more and more conditional logic

    setOn(prev => !prev);
    setClickCount(c => c + 1);
    onToggle?.(!on);
  }

  return <button onClick={handleToggle}>{on ? 'ON' : 'OFF'}</button>;
}

Every new consumer requirement means modifying the component's internals. This violates the Open-Closed Principle and makes the component fragile and hard to test.

How It Works

The Solution

Accept a stateReducer Prop

Instead of adding a new prop for every use case, accept a single stateReducer function. The component computes its default next state, then passes it through the consumer's reducer, which can modify or block the change. One prop to rule them all.

Step 1: Build the useToggle Hook with State Reducer

// Define action types so consumers can react to specific changes
const actionTypes = {
  TOGGLE: 'TOGGLE',
  RESET: 'RESET',
};

// The internal (default) reducer — handles normal behavior
function toggleReducer(state, action) {
  switch (action.type) {
    case actionTypes.TOGGLE:
      return { on: !state.on };
    case actionTypes.RESET:
      return { on: action.initialState };
    default:
      throw new Error(`Unhandled type: ${action.type}`);
  }
}

// The hook that implements the state reducer pattern
function useToggle({
  initialOn = false,
  // The consumer can pass their own reducer to intercept changes
  reducer = toggleReducer,
} = {}) {
  const { current: initialState } = useRef({ on: initialOn });

  // Use the consumer's reducer (which defaults to the internal one)
  const [state, dispatch] = useReducer(reducer, initialState);

  const toggle = () => dispatch({ type: actionTypes.TOGGLE });
  const reset = () => dispatch({ type: actionTypes.RESET, initialState });

  return { on: state.on, toggle, reset, dispatch };
}

// Export action types so consumers can use them
useToggle.actionTypes = actionTypes;
useToggle.reducer = toggleReducer;

Step 2: Consumer Limits Toggles to 4 Clicks

function App() {
  const [clickCount, setClickCount] = useState(0);
  const tooManyClicks = clickCount >= 4;

  // The consumer's custom reducer wraps the default one
  function toggleStateReducer(state, action) {
    // Call the default reducer to get what WOULD happen
    const changes = useToggle.reducer(state, action);

    // Then decide: allow it, modify it, or block it
    if (action.type === useToggle.actionTypes.TOGGLE && tooManyClicks) {
      // Block the change — return current state unchanged
      return state;
    }

    // Otherwise, allow the default behavior
    return changes;
  }

  const { on, toggle, reset } = useToggle({
    reducer: toggleStateReducer,
  });

  return (
    <div>
      <button
        onClick={() => {
          toggle();
          setClickCount(c => c + 1);
        }}
        disabled={tooManyClicks}
      >
        {on ? 'ON' : 'OFF'} ({clickCount} clicks)
      </button>

      <button onClick={() => { reset(); setClickCount(0); }}>
        Reset
      </button>

      {tooManyClicks && (
        <p>You've used all 4 toggles. Reset to continue.</p>
      )}
    </div>
  );
}

Notice how the consumer didn't need a maxClicks prop. They simply wrote a reducer that intercepts the TOGGLE action and blocks it when the limit is reached. Any other consumer could write a completely different reducer for their own needs — without touching the useToggle source code.

Real-World Inspiration

This pattern was popularized by Kent C. Dodds and is used extensively in the Downshift library — a highly flexible autocomplete/select component. Downshift uses a state reducer to let consumers customize dropdown behavior, selection logic, and keyboard interactions without forking the library.

When to Use

Use When

  • Building highly reusable components or hooks
  • Consumers need to customize state transitions
  • You find yourself adding many boolean/callback props for edge cases
  • Publishing a library where you can't predict every use case

Avoid When

  • The component is simple with few states
  • It's an internal component not meant for reuse
  • A simple callback prop covers all known use cases
  • You have fewer than 2-3 consumers with different needs

Key Takeaway

The State Reducer pattern is about inversion of control — instead of anticipating every use case, you hand control to the consumer. It's the ultimate flexibility pattern. Your component defines the default behavior; the consumer decides what actually happens.
