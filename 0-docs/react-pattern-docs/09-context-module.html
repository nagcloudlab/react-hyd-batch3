Performance Pattern

Context Module Functions

Co-locate context helpers with the context itself to keep consumers clean and enable tree-shaking

The Problem

Scattered Dispatch Logic

When using React Context with useReducer, every consumer must know the exact action type strings and payload shapes. This means duplicated logic, fragile string constants, and tight coupling between your context internals and every component that consumes it.

Imagine a shopping cart context consumed by multiple components. Each one constructs its own dispatch calls:

ProductCard.jsx
function ProductCard({ product }) {
  const { dispatch } = useCart();

  return (
    <button onClick={() =>
      dispatch({
        type: 'ADD_ITEM',           // string typo risk
        payload: { id: product.id, name: product.name, price: product.price }
      })
    }>
      Add to Cart
    </button>
  );
}
CartSidebar.jsx
function CartSidebar() {
  const { state, dispatch } = useCart();

  const handleRemove = (itemId) => {
    dispatch({ type: 'REMOVE_ITEM', payload: { id: itemId } });
  };

  const handleQuantity = (itemId, qty) => {
    dispatch({
      type: 'UPDATE_QUANTITY',    // was it UPDATE_QUANTITY or SET_QUANTITY?
      payload: { id: itemId, quantity: qty }
    });
  };

  const handleClear = () => {
    dispatch({ type: 'CLEAR_CART' }); // duplicated across components
  };

  // ... render logic
}
CheckoutPage.jsx
function CheckoutPage() {
  const { state, dispatch } = useCart();

  const handleCheckout = async () => {
    await submitOrder(state.items);
    dispatch({ type: 'CLEAR_CART' }); // same logic, repeated again
  };

  // ... render logic
}

Every consumer is tightly coupled to internal implementation details: action type strings, payload shapes, and dispatch mechanics. Change an action type, and you must update every file.

Architecture Comparison

The Solution

Co-locate Helper Functions with Context

Create a context module that exports everything consumers need: the Provider, a custom hook, and named helper functions that encapsulate all dispatch logic. Consumers never touch raw dispatch or action types directly.

Module Structure

// cart-context.js exports:
CartProvider -- the context provider wrapping useReducer
useCart -- custom hook to consume state + dispatch
addItem(dispatch, item) -- helper to add an item
removeItem(dispatch, itemId) -- helper to remove an item
updateQuantity(dispatch, itemId, qty) -- helper to update quantity
clearCart(dispatch) -- helper to clear the cart

Step 1: Build the Context Module

cart-context.js
import { createContext, useContext, useReducer } from 'react';

// ── Private: action types never leave this file ──
const ActionTypes = {
  ADD_ITEM:        'ADD_ITEM',
  REMOVE_ITEM:     'REMOVE_ITEM',
  UPDATE_QUANTITY: 'UPDATE_QUANTITY',
  CLEAR_CART:      'CLEAR_CART',
};

// ── Private: reducer stays internal ──
function cartReducer(state, action) {
  switch (action.type) {
    case ActionTypes.ADD_ITEM:
      return { ...state, items: [...state.items, action.payload] };
    case ActionTypes.REMOVE_ITEM:
      return { ...state, items: state.items.filter(i => i.id !== action.payload.id) };
    case ActionTypes.UPDATE_QUANTITY:
      return {
        ...state,
        items: state.items.map(i =>
          i.id === action.payload.id ? { ...i, quantity: action.payload.quantity } : i
        ),
      };
    case ActionTypes.CLEAR_CART:
      return { ...state, items: [] };
    default:
      throw new Error(`Unhandled action: ${action.type}`);
  }
}

// ── Context + Provider ──
const CartContext = createContext();

function CartProvider({ children }) {
  const [state, dispatch] = useReducer(cartReducer, { items: [] });
  const value = { state, dispatch };
  return <CartContext.Provider value={value}>{children}</CartContext.Provider>;
}

// ── Custom hook ──
function useCart() {
  const context = useContext(CartContext);
  if (!context) {
    throw new Error('useCart must be used within a CartProvider');
  }
  return context;
}

// ── Context Module Functions (the key pattern!) ──
// Each helper takes dispatch as its first argument.
// This makes them pure, testable, and tree-shakeable.

function addItem(dispatch, item) {
  dispatch({ type: ActionTypes.ADD_ITEM, payload: item });
}

function removeItem(dispatch, itemId) {
  dispatch({ type: ActionTypes.REMOVE_ITEM, payload: { id: itemId } });
}

function updateQuantity(dispatch, itemId, quantity) {
  dispatch({ type: ActionTypes.UPDATE_QUANTITY, payload: { id: itemId, quantity } });
}

function clearCart(dispatch) {
  dispatch({ type: ActionTypes.CLEAR_CART });
}

// ── Public API: everything the consumers need ──
export { CartProvider, useCart, addItem, removeItem, updateQuantity, clearCart };

Step 2: Clean Consumer Components

ProductCard.jsx
import { useCart, addItem } from './cart-context';

function ProductCard({ product }) {
  const { dispatch } = useCart();

  return (
    <div className="product-card">
      <h3>{product.name}</h3>
      <p>${product.price}</p>
      <button onClick={() => addItem(dispatch, product)}>
        Add to Cart
      </button>
    </div>
  );
}
CartSidebar.jsx
import { useCart, removeItem, updateQuantity, clearCart } from './cart-context';

function CartSidebar() {
  const { state, dispatch } = useCart();

  return (
    <aside>
      {state.items.map(item => (
        <div key={item.id}>
          <span>{item.name} x{item.quantity}</span>
          <button onClick={() => updateQuantity(dispatch, item.id, item.quantity + 1)}>+</button>
          <button onClick={() => removeItem(dispatch, item.id)}>Remove</button>
        </div>
      ))}
      <button onClick={() => clearCart(dispatch)}>Clear Cart</button>
    </aside>
  );
}
CheckoutPage.jsx
import { useCart, clearCart } from './cart-context';
// Only imports what it needs -- tree-shaking removes the rest!

function CheckoutPage() {
  const { state, dispatch } = useCart();

  const handleCheckout = async () => {
    await submitOrder(state.items);
    clearCart(dispatch);  // clean, intention-revealing call
  };

  return <button onClick={handleCheckout}>Place Order</button>;
}

Minimal Example: Counter

The same pattern works at any scale. Here is the simplest possible context module:

counter-context.js
import { createContext, useContext, useReducer } from 'react';

const CounterContext = createContext();

function counterReducer(state, action) {
  switch (action.type) {
    case 'INCREMENT': return { count: state.count + 1 };
    case 'DECREMENT': return { count: state.count - 1 };
    case 'RESET':     return { count: 0 };
    default: throw new Error(`Unhandled: ${action.type}`);
  }
}

function CounterProvider({ children }) {
  const [state, dispatch] = useReducer(counterReducer, { count: 0 });
  return (
    <CounterContext.Provider value={{ state, dispatch }}>
      {children}
    </CounterContext.Provider>
  );
}

function useCounter() {
  const ctx = useContext(CounterContext);
  if (!ctx) throw new Error('useCounter must be within CounterProvider');
  return ctx;
}

// Context Module Functions
const increment = (dispatch) => dispatch({ type: 'INCREMENT' });
const decrement = (dispatch) => dispatch({ type: 'DECREMENT' });
const reset     = (dispatch) => dispatch({ type: 'RESET' });

export { CounterProvider, useCounter, increment, decrement, reset };
App.jsx
import { CounterProvider, useCounter, increment, decrement, reset } from './counter-context';

function Counter() {
  const { state, dispatch } = useCounter();

  return (
    <div>
      <p>Count: {state.count}</p>
      <button onClick={() => increment(dispatch)}>+</button>
      <button onClick={() => decrement(dispatch)}>-</button>
      <button onClick={() => reset(dispatch)}>Reset</button>
    </div>
  );
}

function App() {
  return (
    <CounterProvider>
      <Counter />
    </CounterProvider>
  );
}

Why This Pattern Works

Single Source of Truth

Action types and payload shapes live in exactly one file. Change them once, and every consumer stays in sync.

Typo-Proof

Consumers never type string action names. Misspelling addItem gives a clear import error, not a silent runtime bug.

Tree-Shakeable

Because helpers are named exports, bundlers can drop any function a consumer does not import. Smaller bundles for free.

Easy to Test

Each helper is a pure function: pass a mock dispatch, assert the right action was dispatched. No rendering needed.

Colocation Principle

Related things stay together. The context, reducer, hook, and actions all live in one module -- easy to find and reason about.

Async-Ready

Helpers can be async and dispatch multiple actions (loading, success, error) without leaking that logic into components.

When to Use

Use When

  • Your context uses useReducer with multiple action types
  • Multiple components dispatch the same actions
  • Actions involve complex payload construction or async logic
  • You want to test dispatch logic independently of UI

Skip When

  • Your context holds simple state (e.g., a theme toggle with useState)
  • Consumers only read context values and never dispatch
  • There is only one action type -- the overhead is not worth it
  • You are already using a state management library like Zustand or Redux Toolkit

Testing Context Module Functions

Because helpers are plain functions that accept dispatch as an argument, they are trivially testable without rendering any component:

cart-context.test.js
import { addItem, removeItem, clearCart } from './cart-context';

test('addItem dispatches ADD_ITEM with the given product', () => {
  const dispatch = jest.fn();
  const product = { id: '1', name: 'Widget', price: 9.99 };

  addItem(dispatch, product);

  expect(dispatch).toHaveBeenCalledWith({
    type: 'ADD_ITEM',
    payload: product,
  });
});

test('clearCart dispatches CLEAR_CART', () => {
  const dispatch = jest.fn();
  clearCart(dispatch);
  expect(dispatch).toHaveBeenCalledWith({ type: 'CLEAR_CART' });
});

Key Takeaway

Keep related things together. By co-locating dispatch helpers with the context, you create a clean, self-contained module that is easy to use, test, and maintain. Consumers get an intention-revealing API, your action types stay private, and your bundle stays lean through tree-shaking.
