Component Pattern

Render Props

Share behavior between components using a prop whose value is a function

The Problem

Duplicated Logic Across Components

You have reusable logic — mouse tracking, data fetching, form handling, intersection observation — that multiple components need. Without a sharing mechanism, you copy and paste the same useState + useEffect setup into every component that needs it. The logic and the UI become tightly coupled.

Before — Duplicated Mouse Tracking 
// Component A: duplicates mouse tracking logic
function MouseCoordinates() {
  const [position, setPosition] = useState({ x: 0, y: 0 });

  useEffect(() => {
    const handleMove = (e) => {
      setPosition({ x: e.clientX, y: e.clientY });
    };
    window.addEventListener('mousemove', handleMove);
    return () => window.removeEventListener('mousemove', handleMove);
  }, []);

  return (
    <p>Mouse: {position.x}, {position.y}</p>
  );
}

// Component B: exact same logic, different UI
function MouseTrail() {
  const [position, setPosition] = useState({ x: 0, y: 0 });

  useEffect(() => {
    const handleMove = (e) => {
      setPosition({ x: e.clientX, y: e.clientY });
    };
    window.addEventListener('mousemove', handleMove);
    return () => window.removeEventListener('mousemove', handleMove);
  }, []);

  return (
    <div
      style={{
        width: 20, height: 20, borderRadius: '50%',
        background: '#8b5cf6',
        position: 'fixed',
        left: position.x - 10,
        top: position.y - 10,
      }}
    />
  );
}

The mouse-tracking logic is identical in both components. If we need to track the mouse in a third component, we copy it again. Bugs fixed in one copy may not be fixed in others. The logic cannot be tested in isolation from the UI.

How It Works

The Solution

Render Prop Pattern

Extract the reusable logic into a component that accepts a function as a prop (or as children). The component calls that function with the shared state, letting the consumer decide what to render. Logic is written once, UI is flexible.

After — MouseTracker with Render Prop 
import { useState, useEffect } from 'react';

// Reusable logic component — no UI opinion
function MouseTracker({ children }) {
  const [position, setPosition] = useState({ x: 0, y: 0 });

  useEffect(() => {
    const handleMove = (e) => {
      setPosition({ x: e.clientX, y: e.clientY });
    };
    window.addEventListener('mousemove', handleMove);
    return () => window.removeEventListener('mousemove', handleMove);
  }, []);

  // Call children as a function, passing the state
  return children(position);
}
After — Both Components, Zero Duplication 
// Component A: coordinates display
function App() {
  return (
    <MouseTracker>
      {({ x, y }) => (
        <p>Mouse: {x}, {y}</p>
      )}
    </MouseTracker>
  );
}

// Component B: cursor trail — same logic, different UI
function TrailApp() {
  return (
    <MouseTracker>
      {({ x, y }) => (
        <div
          style={{
            width: 20, height: 20,
            borderRadius: '50%',
            background: '#8b5cf6',
            position: 'fixed',
            left: x - 10,
            top: y - 10,
          }}
        />
      )}
    </MouseTracker>
  );
}

// You can also use a "render" prop instead of children:
// <MouseTracker render={({ x, y }) => <p>{x}, {y}</p>} />

Historical Context

While custom hooks have largely replaced render props for logic sharing, render props are still valuable for component composition and inversion of control. Libraries like React Router, Downshift, and Formik used this pattern extensively. Understanding render props deepens your understanding of React's composition model and helps you read older codebases.

When to Use & When Not to Use

When to Use

  • Sharing behavior with completely flexible rendering
  • Building component libraries where consumers control the UI
  • When you need to share logic and wrap JSX around the consumer's output
  • Dynamic composition at render time (the function can vary)

When NOT to Use

  • When a custom hook would be simpler (most cases today)
  • When nesting multiple render props creates "callback hell"
  • For simple state sharing where Context is more appropriate
  • When the shared logic doesn't need to wrap or inject into the UI tree

Render Props vs. Custom Hooks

Render Props

  • Can wrap the consumer's JSX in additional elements
  • Works in class components
  • Can lead to nesting ("wrapper hell")
  • The provider is a component in the tree

Custom Hooks

  • Cleaner, flatter code
  • Easier to compose multiple hooks
  • Only works in function components
  • Cannot wrap JSX — purely logic

Key Takeaway

Render Props invert control — the component with the logic doesn't decide what to render, the consumer does. This creates maximum flexibility: one piece of logic, unlimited visual representations.
