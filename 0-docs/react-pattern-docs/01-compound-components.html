Component Pattern

Compound Components

Components that work together to form a complete UI with implicit shared state

The Problem

Prop Drilling & Rigid APIs

Building a component like an Accordion, Tabs, or Select where multiple sub-components need to share state quickly becomes painful. You end up passing state, handlers, and configuration through every level of the tree. The API becomes rigid and difficult to extend.

Before — Prop Drilling Mess 
// The consumer has to wire everything manually
function App() {
  const [openIndex, setOpenIndex] = useState(null);

  return (
    <Accordion
      openIndex={openIndex}
      onToggle={setOpenIndex}
      items={[
        {
          title: "Section 1",
          content: "Content for section 1",
          isOpen: openIndex === 0,
          onToggle: () => setOpenIndex(openIndex === 0 ? null : 0),
        },
        {
          title: "Section 2",
          content: "Content for section 2",
          isOpen: openIndex === 1,
          onToggle: () => setOpenIndex(openIndex === 1 ? null : 1),
        },
      ]}
    />
  );
}

// Inside Accordion — props are drilled to every item
function Accordion({ items, openIndex, onToggle }) {
  return (
    <div>
      {items.map((item, i) => (
        <div key={i}>
          <button onClick={item.onToggle}>{item.title}</button>
          {item.isOpen && <div>{item.content}</div>}
        </div>
      ))}
    </div>
  );
}

This approach forces the consumer to manage state, compute derived values, and wire callbacks for every single item. Adding features (disabled items, icons, animations) means adding more props to an already bloated configuration object.

How It Works

The Solution

Compound Components with Context

Create a parent component that owns the shared state and exposes it through React.createContext. Child components consume that context implicitly, so the consumer never has to wire state manually. The result is an API that reads like native HTML: <select> and <option>.

After — Implementation 
import { createContext, useContext, useState } from 'react';

// 1. Create a context for the shared state
const AccordionContext = createContext();

function useAccordion() {
  const ctx = useContext(AccordionContext);
  if (!ctx) throw new Error('Accordion compound components must be rendered inside <Accordion>');
  return ctx;
}

// 2. Parent component owns state, provides context
function Accordion({ children }) {
  const [openIndex, setOpenIndex] = useState(null);

  const toggle = (index) => {
    setOpenIndex(prev => (prev === index ? null : index));
  };

  return (
    <AccordionContext.Provider value={{ openIndex, toggle }}>
      <div className="accordion">{children}</div>
    </AccordionContext.Provider>
  );
}

// 3. Child component consumes context implicitly
function AccordionItem({ index, title, children }) {
  const { openIndex, toggle } = useAccordion();
  const isOpen = openIndex === index;

  return (
    <div>
      <button onClick={() => toggle(index)}>
        {title} {isOpen ? '▲' : '▼'}
      </button>
      {isOpen && <div className="panel">{children}</div>}
    </div>
  );
}

// Attach sub-components for a clean namespace
Accordion.Item = AccordionItem;
After — Usage (Clean Consumer API) 
function App() {
  return (
    <Accordion>
      <Accordion.Item index={0} title="Section 1">
        <p>Content for section 1</p>
      </Accordion.Item>
      <Accordion.Item index={1} title="Section 2">
        <p>Content for section 2</p>
      </Accordion.Item>
      <Accordion.Item index={2} title="Section 3">
        <p>Content for section 3</p>
      </Accordion.Item>
    </Accordion>
  );
}

When to Use & When Not to Use

When to Use

  • Complex UI components with related sub-components (Tabs, Accordion, Menu, Dropdown)
  • When you want an expressive, declarative API for consumers
  • When children need shared state but should remain composable
  • Form components with shared validation context

When NOT to Use

  • Simple components with only 1–2 props
  • Components that don't have meaningful sub-components
  • When the added complexity of Context isn't justified
  • When children are truly independent and don't share state

Key Takeaway

Compound Components give you an expressive API like HTML's <select> / <option> — the parent manages state, children consume it implicitly. The consumer writes declarative markup instead of wiring callbacks and state.

Real-World Examples

  • Reach UI — <Menu>, <MenuButton>, <MenuList>, <MenuItem>
  • Radix UI — Every primitive uses compound components (Dialog.Root, Dialog.Trigger, Dialog.Content)
  • Headless UI — <Disclosure>, <Disclosure.Button>, <Disclosure.Panel>
  • Chakra UI — <Tabs>, <TabList>, <Tab>, <TabPanels>
