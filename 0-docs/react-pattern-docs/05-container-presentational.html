Logic Pattern

Container & Presentational

Separate what a component does from how it looks

The Problem

Components That Do Everything

When a single component handles data fetching, state management, business logic, and rendering, it becomes difficult to test, hard to reuse, and painful to modify. You cannot render the UI with mock data, nor can you reuse the data-fetching logic with a different layout.

Before: Mixed Data + UI in One Component

import { useState, useEffect } from 'react';

function UserList() {
  const [users, setUsers] = useState([]);
  const [loading, setLoading] = useState(true);
  const [error, setError] = useState(null);

  // Data logic
  useEffect(() => {
    fetch('/api/users')
      .then(res => {
        if (!res.ok) throw new Error('Failed to fetch');
        return res.json();
      })
      .then(data => setUsers(data))
      .catch(err => setError(err.message))
      .finally(() => setLoading(false));
  }, []);

  // Rendering logic — tightly coupled to the data logic above
  if (loading) return <div className="spinner">Loading...</div>;
  if (error) return <div className="error">{error}</div>;

  return (
    <ul className="user-list">
      {users.map(user => (
        <li key={user.id} className="user-card">
          <img src={user.avatar} alt={user.name} />
          <h3>{user.name}</h3>
          <p>{user.email}</p>
        </li>
      ))}
    </ul>
  );
}

This component cannot be reused with a different data source. You cannot render the user list in Storybook without mocking fetch. Testing requires both API mocking and DOM assertions in the same test.

The Separation

The Solution

Split Into Container + Presentational

The container handles the "what" (data fetching, state, side effects). The presentational handles the "how" (markup, styles, layout). They communicate through props.

After: Presentational Component (Pure UI)

// --- UserList.jsx (Presentational) ---
// Only concerned with rendering. Receives everything via props.

function UserList({ users, loading, error }) {
  if (loading) return <div className="spinner">Loading...</div>;
  if (error) return <div className="error">{error}</div>;

  return (
    <ul className="user-list">
      {users.map(user => (
        <li key={user.id} className="user-card">
          <img src={user.avatar} alt={user.name} />
          <h3>{user.name}</h3>
          <p>{user.email}</p>
        </li>
      ))}
    </ul>
  );
}

// Now you can:
//   - Render <UserList users={mockData} /> in Storybook
//   - Test the UI without mocking fetch
//   - Reuse this with any data source

After: Container Component (Data Logic)

// --- UserListContainer.jsx (Container) ---
// Only concerned with data. Renders the presentational component.

import { useState, useEffect } from 'react';
import UserList from './UserList';

function UserListContainer() {
  const [users, setUsers] = useState([]);
  const [loading, setLoading] = useState(true);
  const [error, setError] = useState(null);

  useEffect(() => {
    fetch('/api/users')
      .then(res => {
        if (!res.ok) throw new Error('Failed to fetch');
        return res.json();
      })
      .then(data => setUsers(data))
      .catch(err => setError(err.message))
      .finally(() => setLoading(false));
  }, []);

  return <UserList users={users} loading={loading} error={error} />;
}

Modern Alternative: Custom Hook + Presentational

Today, the preferred approach is to extract the data logic into a custom hook instead of a container component. The result is the same separation of concerns, but without the extra component layer.

// --- useUsers.js (Custom Hook replaces the Container) ---
import { useState, useEffect } from 'react';

export function useUsers() {
  const [users, setUsers] = useState([]);
  const [loading, setLoading] = useState(true);
  const [error, setError] = useState(null);

  useEffect(() => {
    fetch('/api/users')
      .then(res => {
        if (!res.ok) throw new Error('Failed to fetch');
        return res.json();
      })
      .then(data => setUsers(data))
      .catch(err => setError(err.message))
      .finally(() => setLoading(false));
  }, []);

  return { users, loading, error };
}

// --- UserListPage.jsx (Hook + Presentational together) ---
import { useUsers } from './useUsers';
import UserList from './UserList';

function UserListPage() {
  const { users, loading, error } = useUsers();
  return <UserList users={users} loading={loading} error={error} />;
}

// Same separation, one fewer component in the tree.

A note on history: Dan Abramov, who popularized this pattern, later wrote that he no longer recommends splitting components into containers and presentationals for the sake of it. With Hooks, you can often achieve the same clean separation using a custom hook. The mental model — separating "what" from "how" — is still valuable, even if the implementation technique has evolved.

When to Use

Use This Pattern When

  • A component has grown complex with mixed data logic and rendering
  • You need to reuse the same UI with different data sources
  • You want to test rendering independently from data fetching
  • You're building a design system or component library where pure presentational components are essential

Avoid This Pattern When

  • The component is simple — splitting it adds unnecessary indirection
  • A custom hook achieves the same separation more cleanly (prefer hooks in modern React)
  • You're splitting just because of a naming convention, not because there's a real benefit

Key Takeaway

The real lesson is not about containers vs presentational components — it is about separating concerns. Whether you use this pattern, custom hooks, or another approach, keep your data logic and rendering logic distinct. Your future self (and your teammates) will thank you when they need to change one without breaking the other.
